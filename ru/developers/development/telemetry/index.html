Интеграция с OpenTelemetry

Построенная на OpenTelemetry  — нейтральном к вендорам, отраслевом стандарте фреймворке наблюдаемости — система наблюдаемости Qwen Code предоставляет:

  • Универсальная совместимость: экспорт в любой бэкенд OpenTelemetry (Aliyun, Jaeger, Prometheus, Datadog и т.д.)
  • Стандартизированные данные: используйте единые форматы и методы сбора в вашем инструментарии
  • Перспективная интеграция: подключайтесь к существующей и будущей инфраструктуре наблюдаемости
  • Отсутствие привязки к вендору: переключайтесь между бэкендами без изменения вашего инструментирования

Конфигурация

Всё поведение телеметрии управляется через файл .qwen/settings.json. Эти настройки могут быть переопределены переменными окружения или флагами CLI.

Примечание о булевых переменных окружения: для булевых настроек (enabled, logPrompts, includeSensitiveSpanAttributes) установка соответствующей переменной окружения в true или 1 включит функцию. Любое другое значение отключит её.

Примечание о целочисленных переменных окружения: QWEN_TELEMETRY_SENSITIVE_SPAN_ATTRIBUTE_MAX_LENGTH должна быть положительным целым числом при установке. Некорректные значения приводят к ошибке разрешения конфигурации телеметрии, а не к бесшумному возврату к значению по умолчанию.

Чувствительные атрибуты спанов: когда includeSensitiveSpanAttributes включён, происходят две вещи:

  1. Собственные атрибуты спанов (qwen-code.interaction, api.generateContent*, tool.<name>) содержат дословное содержимое диалога:

    • Пользовательские промпты (new_context)
    • Системные промпты (system_prompt — полный текст один раз за сессию, дедуплицирован по хешу SHA-256; последующие спаны несут только system_prompt_hash + system_prompt_preview + system_prompt_length)
    • Схемы инструментов (испускаются как события tool_schema, также дедуплицируются по хешу)
    • Входные данные инструментов (tool_input) и результаты инструментов (tool_result)
    • Вывод модели (response.model_output)

    Каждая полезная нагрузка обрезается до sensitiveSpanAttributeMaxLength единиц строки JavaScript. По умолчанию 1 МиБ (1048576), увеличено с предыдущего значения 60 КиБ; установите 61440, чтобы сохранить старый предел. Ограничение должно быть от 1 до 104857600 (100 МиБ). Для именованных атрибутов фиксированные метки, такие как [USER PROMPT], [TOOL INPUT: ...] и [TOOL RESULT: ...], учитываются в лимите; маркер обрезки также учитывается в нём. Лимит измеряется как длина строки JavaScript, а не байт UTF-8. Поэтому не-ASCII контент может занимать больше байт после экспорта OTLP. Для большинства типов полезной нагрузки обрезка добавляет как *_truncated, так и *_original_length. Системные промпты также устанавливают system_prompt_truncated при обрезке, но используют всегда присутствующий system_prompt_length для исходной длины.

  2. Спаны моста лог->спан (используются, когда HTTP-трейсы экспортируются без эндпоинта логов) сохраняют свои существующие поля prompt, function_args и response_text вместо того, чтобы быть отброшенными.

⚠️ Предупреждение безопасности: включение этого флага передаёт полную историю диалога, содержимое файлов, прочитанных read_file, команды оболочки и их вывод (включая секреты в переменных окружения или аргументах) и ответы модели в настроенный OTLP бэкенд. Относитесь к бэкенду как к привилегированному приёмнику данных. Флаг по умолчанию false.

Стоимость / размер полезной нагрузки: тяжёлый вызов при лимите по умолчанию (1 МиБ системный промпт плюс 10 вызовов инструментов, каждый до 1 МиБ ввода + 1 МиБ результата, плюс 1 МиБ вывода модели) может дать до ~22 МиБ полезной нагрузки атрибутов до сжатия OTLP, плюс до 1 МиБ на каждую испущенную схему инструмента в рабочих областях с большими определениями инструментов. Это ограничение на стороне приложения Qwen Code, а не гарантия того, что каждый коллектор или бэкенд принимает один атрибут такого размера. Если спаны отвергаются или отбрасываются, уменьшите sensitiveSpanAttributeMaxLength (например, до 61440) и отслеживайте пропускную способность экспортера.

Эта настройка не отключает чувствительные данные в OTel-логах или других источниках телеметрии; телеметрия ответов от не внутренних API может заполнять response_text, поэтому OTel-логи, телеметрия UI и запись чата могут получать текст ответа независимо от этой настройки. QwenLogger не включает response_text.

Маршрутизация сигналов HTTP OTLP: при использовании протокола HTTP (otlpProtocol: "http") Qwen Code автоматически добавляет пути, специфичные для сигнала (/v1/traces, /v1/logs, /v1/metrics), к базовому otlpEndpoint. Например, http://collector:4318 становится http://collector:4318/v1/traces для трейсов. Если URL уже заканчивается на путь сигнала, он используется как есть. Переопределения эндпоинтов для конкретных сигналов (otlpTracesEndpoint и т.д.) имеют приоритет над базовым эндпоинтом и используются дословно. Протокол gRPC использует маршрутизацию на основе сервисов и не добавляет пути.

Переменные окружения для эндпоинтов отдельных сигналов также принимают стандартные имена OpenTelemetry: OTEL_EXPORTER_OTLP_TRACES_ENDPOINT, OTEL_EXPORTER_OTLP_LOGS_ENDPOINT, OTEL_EXPORTER_OTLP_METRICS_ENDPOINT. Варианты QWEN_TELEMETRY_OTLP_* имеют приоритет над вариантами OTEL_*.

Подробную информацию обо всех опциях конфигурации см. в Руководстве по конфигурации.

Атрибуты ресурса

Атрибуты ресурса — это статические пары ключ-значение, прикрепляемые к каждому спану, логу и метрике, экспортируемым через OTLP. Используйте их для сегментирования телеметрии по команде, окружению, региону развертывания или любому другому измерению, которое имеет значение для вашего бэкенда.

Два источника, объединяемые в порядке приоритета (низший → высший):

  1. Стандартная переменная окружения OTEL_RESOURCE_ATTRIBUTES
  2. telemetry.resourceAttributes в .qwen/settings.json (переопределяет переменную окружения при конфликте ключей)

OTEL_SERVICE_NAME — это отдельный запасной вариант: при установке он переопределяет service.name из любого другого источника (согласно спецификации OpenTelemetry).

Примеры

Сегментировать всю телеметрию по команде/окружению:

export OTEL_RESOURCE_ATTRIBUTES="team=platform,env=prod,cost_center=eng-123"

Направить к коллектору для конкретного клиента через service.name:

export OTEL_SERVICE_NAME=qwen-code-ci

Базовая конфигурация флота (~/.qwen/settings.json) + переопределение на хост:

{
  "telemetry": {
    "resourceAttributes": {
      "deployment.environment": "production",
      "service.namespace": "engineering-tooling"
    }
  }
}
# Добавить разовый тег без изменения настроек:
export OTEL_RESOURCE_ATTRIBUTES="debug_run=true"

Зарезервированные ключи

Некоторые ключи управляются во время выполнения и не могут быть переопределены:

  • service.version — всегда устанавливается в версию запущенного CLI. Установка из любого источника молча игнорируется с предупреждением.
  • session.id — внедряется во время выполнения для каждой сессии. Значения, предоставленные пользователем из переменной окружения или настроек, игнорируются с предупреждением. Причина в том, что атрибуты ресурса автоматически прикрепляются к каждой точке данных метрики; разрешение пользовательского переопределения обошло бы контроли кардинальности ниже. Спаны и логи всегда содержат session.id.

service.name не зарезервирован; он следует цепочке приоритетов выше.

Формат

OTEL_RESOURCE_ATTRIBUTES соответствует спецификации OpenTelemetry: key1=value1,key2=value2 со значениями, закодированными в процентной кодировке. Пробелы в значениях должны быть закодированы как %20, запятые как %2C (незакодированные запятые разбивают значение в неправильном месте, и вторая половина отбрасывается как некорректная). Некорректные пары пропускаются с предупреждением, а не приводят к сбою запуска телеметрии.

Устранение неполадок: когда кажется, что предоставленный пользователем атрибут не вступает в силу

Зарезервированные ключи (service.version, session.id), некорректные пары, нестроковые значения настроек и неверная процентная кодировка молча игнорируются с предупреждением, записываемым через диагностический канал OpenTelemetry. Этот канал направляет в файл отладочного лога (~/.qwen/log/otel-*.log), не на консоль, поэтому поведение может выглядеть как бесшумный сбой.

Если пользовательский атрибут ресурса не появляется в экспортированной телеметрии:

  1. Проверьте ~/.qwen/log/otel-*.log на строки, соответствующие cannot override (зарезервированный ключ отброшен), Skipping malformed (некорректная пара в переменной окружения) или must be a string (нестроковое значение настройки).
  2. Убедитесь, что переменная окружения установлена в окружении процесса qwen-code (не только в вашей оболочке) и что значения закодированы в процентной кодировке.
  3. Подтвердите, что telemetry.enabled установлен в true — инициализация телеметрии выполняется только при включении.

Контроли кардинальности

Метрики агрегируются по набору атрибутов на бэкенде — каждая уникальная комбинация значений атрибутов создаёт новый временной ряд. Прикрепление высококардинального поля, такого как session.id, к метрике вызывает разрастание временных рядов, пропорциональное количеству сессий, что быстро исчерпывает хранилище метрик.

Чтобы предотвратить это, Qwen Code по умолчанию не включает высококардинальные атрибуты в точки данных метрик. Спаны и логи являются по-событийными и не затрагиваются, поэтому они продолжают содержать session.id для корреляции трейсов и логов.

telemetry.metrics.includeSessionId (по умолчанию: false)

Установка этого значения в true (через настройки или QWEN_TELEMETRY_METRICS_INCLUDE_SESSION_ID=true) повторно прикрепляет session.id к каждой точке данных метрики.

⚠️ Предупреждение: каждая сессия CLI создаёт новое значение. Оставление этого включённым для флота взорвёт хранилище метрик. Рекомендуется только для краткосрочной отладки. Для долгосрочной корреляции сессий используйте вместо этого запросы к трейсам или логам.

Миграция с предыдущих версий

До этого релиза session.id по умолчанию прикреплялся к метрикам. Если ваши запросы Prometheus / дашборды Grafana / правила оповещений ссылаются на session_id в метрике, у вас есть два варианта:

Вариант A — восстановить предыдущее поведение для краткосрочной отладки:

export QWEN_TELEMETRY_METRICS_INCLUDE_SESSION_ID=true

или:

{
  "telemetry": {
    "metrics": { "includeSessionId": true }
  }
}

Вариант B (рекомендуется) — переместите анализ на уровне сессий из метрик. Спаны и логи по-прежнему содержат session.id, а бэкенды трейсов и логов (Jaeger, Tempo, Loki, Aliyun SLS / ARMS Tracing) обрабатывают сегментацию по сессиям нативно без нагрузки на кардинальность.

HTTP-спан на стороне клиента для исходящих fetch-запросов

Когда телеметрия включена, Qwen Code регистрирует UndiciInstrumentation, который создаёт HTTP-спан на стороне клиента для каждого исходящего fetch()-запроса, инициированного процессом — включая LLM SDK (openai, @google/genai, @anthropic-ai/sdk), MCP StreamableHTTP-клиент, инструмент WebFetch и любые вызовы вне процесса расширений IDE. Спан позволяет увидеть сетевую задержку (TTFB / передачу тела ответа) отдельно от времени обработки вышестоящей модели, что существующий спан api.generateContent сам по себе не может различить.

Эти спаны отправляются в ваш собственный коллектор OTLP (или файл outfile) так же, как и остальная телеметрия — они не влияют на то, что записывается в сам исходящий HTTP-запрос. Управление тем, записывается ли также заголовок W3C traceparent в исходящий поток запроса, определяется отдельной, важной для безопасности настройкой, описанной в исходящей корреляции ниже.

Предотвращение циклов обратной связи. OTel SDK использует fetch внутри для загрузки данных OTLP. Без защиты инструментирование fetch привело бы к трассировке этих загрузок, которые сами были бы загружены, создавая бесконечный цикл. Инструментирование undici в Qwen Code настроено с ignoreRequestHook, который пропускает URL, соответствующие префиксам настроенных telemetry.otlpEndpoint / telemetry.otlpTracesEndpoint / telemetry.otlpLogsEndpoint / telemetry.otlpMetricsEndpoint. В режиме вывода в файл исходящих HTTP-загрузок нет, поэтому хук ничего не делает.

Исходящая корреляция (ВАЖНО ДЛЯ БЕЗОПАСНОСТИ)

Эти настройки находятся в отдельном пространстве имён верхнего уровня от telemetry.* намеренно: телеметрия управляет потоком данных в собственный бэкенд наблюдаемости оператора, в то время как outboundCorrelation.* управляет тем, какие данные корреляции на стороне клиента Qwen Code записывает в исходящие запросы к LLM API, которые достигают сторонних эндпоинтов LLM-провайдеров (DashScope, OpenAI, Anthropic и т.д.). Разные получатели — разное решение о согласии. Все значения по умолчанию выключены. См. обсуждение в рецензии PR #4390 для обоснования такой структуры.

outboundCorrelation.propagateTraceContext

"outboundCorrelation": {
  "propagateTraceContext": false // по умолчанию
}

Когда false (по умолчанию), Qwen Code устанавливает пустой TextMapPropagator в OTel SDK. UndiciInstrumentation всё ещё создаёт клиентские HTTP-спаны для вашего коллектора OTLP, но propagation.inject() ничего не делает, поэтому никакой traceparent не записывается в исходящие запросы. Идентификаторы трейсов остаются внутренними для коллектора оператора.

Когда true, устанавливается композитный пропагатор W3C по умолчанию SDK (tracecontext + baggage), и стандартный заголовок traceparent записывается в каждый исходящий fetch:

traceparent: 00-<32-символа hex traceId>-<16-символов hex parentSpanId>-<01-sampled | 00-not-sampled>

Включайте только в том случае, если LLM-провайдер также передаёт данные в ваш коллектор OTel для сквозного связывания трейсов между процессами — например, ARMS Tracing, обслуживающий DashScope. Для большинства операторов значение false; межвендорное продолжение трейсов является нишевым. Зависит от telemetry.enabled: true. OTel SDK инициализируется только при включенной телеметрии, поэтому propagateTraceContext работает только в этом состоянии. Установка true при отключенной телеметрии — это молчаливая пустая операция: нет SDK, нет пропагатора, нет traceparent в передаваемых данных. Проверьте оба флага при настройке интеграции ARMS + DashScope:

{
  "telemetry": {
    "enabled": true,
    "otlpTracesEndpoint": "http://tracing-analysis-...",
  },
  "outboundCorrelation": {
    "propagateTraceContext": true,
  },
}

Другие заголовки исходящей корреляции

X-Qwen-Code-Session-Id и X-Qwen-Code-Request-Id не являются частью этого PR. Они будут спроектированы и предложены в своих собственных дальнейших PR в том же пространстве имён outboundCorrelation.*, каждый с собственной моделью угроз и потоком согласия оператора. В ходе ревью PR #4390 (LaZzyMan) был установлен принцип: «объём работы телеметрии не включает отправку идентификаторов провайдерам LLM»; работа над заголовками корреляции переходит в собственное обсуждение дизайна, а не включается в раздел телеметрии.

Экспорт вручную через OTLP

Чтобы просматривать телеметрию Qwen Code в Alibaba Cloud Managed Service for OpenTelemetry, настройте Qwen Code на экспорт в конечную точку OTLP, предоставляемую ARMS.

Установка "target": "gcp" сама по себе не настраивает конечную точку экспорта. Если otlpEndpoint не задан, Qwen Code по-прежнему использует http://localhost:4317. Если задан outfile, он переопределяет otlpEndpoint, и телеметрия записывается в файл, а не отправляется в Alibaba Cloud.

  1. Включите телеметрию в вашем .qwen/settings.json и задайте конечную точку OTLP:

    Вариант A: протокол gRPC (стандартная конечная точка OTLP):

    {
  "telemetry": {
    "enabled": true,
    "target": "gcp",
    "otlpEndpoint": "https://<your-otlp-endpoint>",
    "otlpProtocol": "grpc"
  }
}

    Вариант B: протокол HTTP с раздельными конечными точками для разных сигналов (для бэкендов, использующих нестандартные пути, например, /api/otlp/traces вместо /v1/traces):

    {
  "telemetry": {
    "enabled": true,
    "otlpProtocol": "http",
    "otlpTracesEndpoint": "http://<host>/<token>/api/otlp/traces",
    "otlpLogsEndpoint": "http://<host>/<token>/api/otlp/logs",
    "otlpMetricsEndpoint": "http://<host>/<token>/api/otlp/metrics"
  }
}

    Примечание: При использовании протокола HTTP только с otlpEndpoint (без переопределений для отдельных сигналов) Qwen Code добавляет стандартные пути OTLP (/v1/traces, /v1/logs, /v1/metrics) к базовому URL. Если ваш бэкенд использует другие пути, используйте переопределения для отдельных сигналов, как показано в варианте B.

  2. Если ваша конечная точка Alibaba Cloud требует аутентификации, укажите заголовки OTLP через стандартные переменные окружения OpenTelemetry, такие как OTEL_EXPORTER_OTLP_HEADERS (или варианты для конкретных сигналов). Qwen Code в настоящее время не предоставляет возможность задавать заголовки аутентификации OTLP напрямую в .qwen/settings.json.

  3. Запустите Qwen Code и отправляйте запросы.

  4. Просматривайте телеметрию в Managed Service for OpenTelemetry:

    • Обзор продукта: Что такое Managed Service for OpenTelemetry? 
    • Начало работы: Начало работы с Managed Service for OpenTelemetry 
    • Точки входа в консоль:
      • Материковый Китай: trace.console.aliyun.com  (устаревшая консоль: tracing.console.aliyun.com )
      • Международный регион: arms.console.alibabacloud.com 
    • В консоли используйте раздел Applications для просмотра трейсов и топологии сервисов.
    • Чтобы найти конечную точку OTLP и информацию о доступе:
      • Новая консоль (trace.console.aliyun.com или международная): перейдите в Integration Center.
      • Устаревшая консоль (tracing.console.aliyun.com): перейдите в Cluster Configurations → Access point information.

Локальная телеметрия

Для локальной разработки и отладки вы можете захватывать данные телеметрии локально:

Вывод в файл (рекомендуется)

  1. Включите телеметрию в вашем .qwen/settings.json:

    {
  "telemetry": {
    "enabled": true,
    "outfile": ".qwen/telemetry.log"
  }
}

    Примечание: Когда outfile задан, экспорт через OTLP автоматически отключается. Настройки target и otlpEndpoint не нужны при выводе только в файл и могут быть безопасно опущены в вашей конфигурации.

  2. Запустите Qwen Code и отправляйте запросы.

  3. Просматривайте логи и метрики в указанном файле (например, .qwen/telemetry.log).

Экспорт через коллектор (продвинутый уровень)

  1. Запустите скрипт автоматизации: 
    npm run telemetry -- --target=local
    Это:
    • Скачает и запустит Jaeger и коллектор OTEL
    • Настроит вашу рабочую область для локальной телеметрии
    • Предоставит интерфейс Jaeger по адресу http://localhost:16686 
    • Сохранит логи/метрики в ~/.qwen/tmp/<projectHash>/otel/collector.log
    • Остановит коллектор при выходе (например, Ctrl+C)
  2. Запустите Qwen Code и отправляйте запросы.
  3. Просматривайте трейсы по адресу http://localhost:16686  и логи/метрики в файле лога коллектора.

Логи и метрики

В следующем разделе описывается структура логов и метрик, генерируемых для Qwen Code.

  • sessionId включается как общий атрибут во все логи и метрики.

Логи

Логи — это записи событий с временными метками. Для Qwen Code регистрируются следующие события:

  • qwen-code.config: Это событие происходит один раз при запуске CLI с конфигурацией.

    • Атрибуты:
      • model (строка)
      • sandbox_enabled (булево)
      • core_tools_enabled (строка)
      • approval_mode (строка)
      • file_filtering_respect_git_ignore (булево)
      • debug_mode (булево)
      • truncate_tool_output_threshold (число)
      • truncate_tool_output_lines (число)
      • hooks (строка, разделённые запятыми типы событий хуков, опускается, если хуки отключены)
      • ide_enabled (булево)
      • interactive_shell_enabled (булево)
      • mcp_servers (строка)
      • output_format (строка: “text” или “json”)

  • qwen-code.user_prompt: Это событие происходит, когда пользователь отправляет запрос.

    • Атрибуты:
      • prompt_length (целое)
      • prompt_id (строка)
      • prompt (строка, этот атрибут исключается, если log_prompts_enabled настроено как false)
      • auth_type (строка)

  • qwen-code.tool_call: Это событие происходит для каждого вызова функции.

    • Атрибуты:
      • function_name
      • function_args
      • duration_ms
      • success (булево)
      • decision (строка: “accept”, “reject”, “auto_accept” или “modify”, если применимо)
      • error (если применимо)
      • error_type (если применимо)
      • content_length (целое, если применимо)
      • metadata (если применимо, словарь строка -> любой тип)

  • qwen-code.file_operation: Это событие происходит для каждой операции с файлом.

    • Атрибуты:
      • tool_name (строка)
      • operation (строка: “create”, “read”, “update”)
      • lines (целое, если применимо)
      • mimetype (строка, если применимо)
      • extension (строка, если применимо)
      • programming_language (строка, если применимо)
      • diff_stat (строка JSON, если применимо): JSON-строка со следующими полями:
        • ai_added_lines (целое)
        • ai_removed_lines (целое)
        • user_added_lines (целое)
        • user_removed_lines (целое)

  • qwen-code.api_request: Это событие происходит при выполнении запроса к Qwen API.

    • Атрибуты:
      • model
      • request_text (если применимо)

  • qwen-code.api_error: Это событие происходит, если запрос к API завершился ошибкой.

    • Атрибуты:
      • model
      • error
      • error_type
      • status_code
      • duration_ms
      • auth_type

  • qwen-code.api_response: Это событие происходит при получении ответа от Qwen API.

    • Атрибуты:
      • model
      • status_code
      • duration_ms
      • error (необязательно)
      • input_token_count
      • output_token_count
      • cached_content_token_count
      • thoughts_token_count
      • response_text (если применимо)
      • auth_type

  • qwen-code.tool_output_truncated: Это событие происходит, когда вывод вызова инструмента слишком велик и усекается.

    • Атрибуты:
      • tool_name (строка)
      • original_content_length (целое)
      • truncated_content_length (целое)
      • threshold (целое)
      • lines (целое)
      • prompt_id (строка)

  • qwen-code.malformed_json_response: Это событие происходит, когда ответ generateJson от Qwen API не может быть разобран как JSON.

    • Атрибуты:
      • model

  • qwen-code.flash_fallback: Это событие происходит, когда Qwen Code переключается на Flash как запасной вариант.

    • Атрибуты:
      • auth_type

  • qwen-code.slash_command: Это событие происходит, когда пользователь выполняет слеш-команду.

    • Атрибуты:
      • command (строка)
      • subcommand (строка, если применимо)

  • qwen-code.extension_enable: Это событие происходит, когда расширение включается

  • qwen-code.extension_install: Это событие происходит, когда расширение устанавливается

    • Атрибуты:
      • extension_name (строка)
      • extension_version (строка)
      • extension_source (строка)
      • status (строка)

  • qwen-code.extension_uninstall: Это событие происходит, когда расширение удаляется

Метрики

Метрики — это численные измерения поведения с течением времени. Для Qwen Code собираются следующие метрики (имена метрик остаются qwen-code.* для совместимости):

  • qwen-code.session.count (Counter, Int): Увеличивается на 1 при каждом запуске CLI.

  • qwen-code.tool.call.count (Counter, Int): Считает количество вызовов инструментов.

    • Атрибуты:
      • function_name
      • success (булево)
      • decision (строка: “accept”, “reject” или “modify”, если применимо)
      • tool_type (строка: “mcp” или “native”, если применимо)

  • qwen-code.tool.call.latency (Histogram, ms): Измеряет задержку вызова инструмента.

    • Атрибуты:
      • function_name
      • decision (строка: “accept”, “reject” или “modify”, если применимо)

  • qwen-code.api.request.count (Counter, Int): Считает все запросы к API.

    • Атрибуты:
      • model
      • status_code
      • error_type (если применимо)

  • qwen-code.api.request.latency (Histogram, ms): Измеряет задержку запроса к API.

    • Атрибуты:
      • model

  • qwen-code.token.usage (Counter, Int): Считает количество использованных токенов.

    • Атрибуты:
      • model
      • type (строка: “input”, “output”, “thought” или “cache”)

  • qwen-code.file.operation.count (Counter, Int): Считает операции с файлами.

    • Атрибуты:
      • operation (строка: “create”, “read”, “update”): Тип операции с файлом.
      • lines (Int, если применимо): Количество строк в файле.
      • mimetype (строка, если применимо): MIME-тип файла.
      • extension (строка, если применимо): Расширение файла.
      • model_added_lines (Int, если применимо): Количество строк, добавленных/изменённых моделью.
      • model_removed_lines (Int, если применимо): Количество строк, удалённых/изменённых моделью.
      • user_added_lines (Int, если применимо): Количество строк, добавленных/изменённых пользователем в предложенных ИИ изменениях.
      • user_removed_lines (Int, если применимо): Количество строк, удалённых/изменённых пользователем в предложенных ИИ изменениях.
      • programming_language (строка, если применимо): Язык программирования файла.

  • qwen-code.chat_compression (Counter, Int): Считает операции сжатия чата

    • Атрибуты:
      • tokens_before: (Int): Количество токенов в контексте до сжатия
      • tokens_after: (Int): Количество токенов в контексте после сжатия

Banner 自定义区域设计方案

[!note] Позволяет пользователям заменять ASCII-логотип QWEN, заменять заголовок бренда, полностью скрывать Banner — но не позволяет удалять информацию, необходимую для диагностики и доверия (номер версии, способ аутентификации, модель, рабочую директорию).

Обзор

При запуске Qwen Code CLI в верхней части терминала печатается Banner, содержащий ASCII-логотип QWEN и информационную панель с рамкой. Существует несколько реальных сценариев, требующих контроля над этой областью:

  • Белые метки / интеграция сторонних брендов: при встраивании Qwen Code в корпоративные или командные продукты необходимо показывать собственный бренд вместо стандартного “Qwen Code”.
  • Персонализация: отдельные пользователи хотят привести Terminal Banner в соответствие с командными стандартами или личными предпочтениями.
  • Мультитенантность / различие экземпляров: в совместной среде разные команды хотят быстро идентифицировать, какой экземпляр они используют.

Позиция дизайна очень проста: внешний вид бренда можно заменить; информацию времени выполнения заменять нельзя. Настройка позволяет пользователям только накладывать свой бренд поверх, не допускается скрывать ключевую информацию для диагностики. Каждое решение «можно изменить / нельзя изменить» в данном документе исходит из этой позиции.

Соответствующий issue: #3005 .

Разделение области Banner

Текущий Banner рендерится компонентом Header (монтируется через AppHeader). Общая разбивка:

  marginX=2                                                           marginX=2
  │                                                                          │
  ▼                                                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   ┌──── Столбец логотипа ──┐  gap=2  ┌──── Информационная панель (с рамкой) ┐ │
│   │                      │         │                                     │  │
│   │  ███ QWEN ASCII ███  │         │  ① Заголовок: >_ Qwen Code (vX.Y.Z)│  │
│   │  ███   ART ART  ███  │         │  ② Подзаголовок: «Пустая строка / пользовательское переопределение»│  │
│   │  ███ QWEN ASCII ███  │         │  ③ Состояние: Qwen OAuth | qwen-…    │  │
│   │                      │         │  ④ Путь: ~/projects/example         │  │
│   └──────── A ───────────┘         └──────────────── B ──────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
                              Владелец области: AppHeader
                          Компонент Tips рендерится ниже (контролируется ui.hideTips) │

Два главных блока:

  • A. Столбец логотипа — одиночный ASCII-арт с градиентом. Текущий источник: shortAsciiLogo в packages/cli/src/ui/components/AsciiArt.ts.
  • B. Информационная панель — информационный блок с рамкой, всего четыре строки. Вторая строка по умолчанию — пустой визуальный spacer, опционально заменяется на подзаголовок, переданный вызывающей стороной:
    • B① Заголовок: >_ Qwen Code (vX.Y.Z) — текст бренда + суффикс версии.
    • B② Подзаголовок / spacer: по умолчанию строка с одним пробелом; при установке ui.customBannerSubtitle отображается очищенная однострочная строка подзаголовка (например, форк может использовать Built-in DataWorks Official Skills).
    • B③ Состояние: <тип отображения аутентификации> | <модель> (переключение /model).
    • B④ Путь: рабочая директория, сокращённая с использованием tildeify и усечения.

Внешний компонент <AppHeader> уже обрабатывает полное скрытие Banner в режиме чтения с экрана на основе showBanner = !config.getScreenReader() (в этом случае возвращается к чистому текстовому выводу).

Правила настройки — что можно менять, что заблокировано

ОбластьТекущий источникКатегория настройкиПричина блокировки/открытия
A. Столбец логотипаshortAsciiLogo (AsciiArt.ts)Можно заменить + АвтоскрытиеЧисто брендовая зона. Сценарии белых меток требуют полного контроля визуала. Существующее поведение «автоскрытие логотипа на узких терминалах» сохраняется.
B①. Текст заголовка (>_ Qwen Code)жёстко закодирован в Header.tsxМожно заменитьБрендовая зона. Начальные символы >_ — часть существующего бренда; если они не нужны, пользователь может опустить их в customBannerTitle.
B①. Суффикс версии ((vX.Y.Z))пропс versionЗаблокированНеобходим для диагностики и поддержки. Если его скрыть, ответить на вопрос «какую версию ты используешь?» можно только через --version, что создаёт реальные издержки для поддержки. Мы жертвуем небольшим опытом белых меток в пользу доступности поддержки.
B②. Подзаголовок / строка spacerпо умолчанию пустоМожно заменитьЧисто брендовая / контекстная зона. Форки с белыми метками используют её для маркировки версии сборки (например, “Built-in DataWorks Official Skills”). Правила очистки совпадают с заголовком; допускается только одна строка, разрывы строки, нарушающие макет, не принимаются.
B③. Строка состояния (аутентификация + модель)formattedAuthType, проп modelЗаблокированаОперационный и сигнал безопасности. Пользователь должен видеть, какие учётные данные используются и какая модель фактически потребляет токены. Любое скрытие/замена — это footgun, даже в сценарии белых меток.
B④. Строка пути (рабочая директория)проп workingDirectoryЗаблокированаОперационная информация. «В какой я сейчас директории?» — частый вопрос; Banner — единственный авторитетный ответ.
Весь Banner (A + B)точка монтирования <Header> в AppHeader.tsxМожно скрытьНастройка ui.hideBanner: true пропускает оба блока A и B — аналогично существующему переключателю режима чтения с экрана. <Tips> по-прежнему контролируется отдельным ui.hideTips.

Вышеуказанной матрице соответствуют четыре параметра настройки, и только они:

ПараметрЗначение по умолчаниюЭффектОбласть влияния

Намеренно не предоставляемые возможности:

  • Нет переключателя для «скрыть только суффикс версии».
  • Нет переключателя для «скрыть только строку аутентификации/модели».
  • Нет переключателя для «скрыть только строку пути».
  • Нет возможности изменить цвет градиента логотипа (цвет управляется темой).
  • Нет возможности изменить порядок или структуру информационной панели.

Если в будущем возникнет реальная потребность, она должна быть рассмотрена как новое поле отдельно, а не выводиться из трёх указанных полей.

Руководство по настройке пользователя — как изменить

Обзор ограничений

Каждая настройка banner подчиняется нескольким группам лимитов. Прежде чем писать art вручную, просмотрите их, чтобы парсер не отбросил их молча или не обрезал.

ЭлементЛимит
Количество символов заголовкаМаксимум 80 символов (после очистки). При превышении обрезается и выдаётся предупреждение [BANNER]. Символы перевода строки и управляющие символы удаляются до подсчёта.
Количество символов подзаголовкаМаксимум 160 символов (после очистки). Конвейер очистки совпадает с заголовком; при превышении также выводится предупреждение [BANNER].
Размер блока ASCII artМаксимум 200 строк × 200 столбцов на один уровень. При превышении обрезается и выдаётся предупреждение [BANNER].
Размер файла ASCII artМаксимум 64 КБ. Если файл больше, читаются только первые 64 КБ, остальное игнорируется.
Фактическая отображаемая ширина ASCII artОпределяется количеством столбцов терминала при запуске, не фиксированным числом символов. Формула и доступные значения для различных ширин терминалов см. ниже в разделе «Насколько большим может быть логотип? — Бюджет ширины».

Для ASCII art нет фиксированного лимита количества символов — есть только указанные выше жёсткие лимиты (строки/столбцы) и бюджет ширины, вычисляемый при запуске по количеству столбцов терминала. То же самое имя бренда из 17 символов может поместиться в одну строку или нет при смене шрифта — зависит от визуальной ширины, а не от количества букв.

Расположение конфигурации

Все четыре параметра находятся в узле ui файла settings.json. Поддерживаются как пользовательский уровень (~/.qwen/settings.json), так и уровень рабочей области (.qwen/settings.json в корне проекта), с приоритетом слияния по стандарту (workspace переопределяет user, system переопределяет workspace).

customAsciiArt является исключением: парсер не рассматривает весь объект как единое значение, заменяемое целиком в scope более высокого приоритета, а проходит по уровням, пересекая все scope. Если пользовательская настройка определяет { small }, а настройка рабочей области — { large }, то оба вступают в силу — small берётся из user, large — из workspace. Это позволяет одновременно выполнить две задачи:

  1. Каждый элемент { path } разрешается относительно файла, в котором он объявлен (рабочая область .qwen/ vs. пользователь ~/.qwen/); при просмотре только объединённого вида теряется информация о scope.
  2. Пользователь может оставить уровень large по умолчанию в личных настройках, а в рабочей области переопределить только small, не переписывая каждый раз весь объект.

Если один и тот же уровень определён в нескольких scope, применяется обычный приоритет (system > workspace > user). Если в любом scope установить customAsciiArt как строку или { path }, он также заполнит оба уровня этого scope.

Полное скрытие Banner

{
  "ui": {
    "hideBanner": true,
  },
}

При запуске будут пропущены столбец логотипа и информационная панель. Если также не задан ui.hideTips, подсказки (Tips) всё ещё будут отображаться.

Замена заголовка бренда

{
  "ui": {
    "customBannerTitle": "Acme CLI",
  },
}

Информационная панель будет отображать Acme CLI (vX.Y.Z). При установке пользовательского заголовка символы >_ по умолчанию не включаются; чтобы их сохранить, напишите: "customBannerTitle": ">_ Acme CLI".

Добавление подзаголовка бренда

{
  "ui": {
    "customBannerSubtitle": "Built-in DataWorks Official Skills",
  },
}

Подзаголовок появится отдельной строкой второстепенным цветом текста, заменяя пустую строку spacer по умолчанию (то есть строку, которая находится между заголовком и строкой аутентификации/модели):

┌─────────────────────────────────────────────────────────┐
│ DataWorks DataAgent (vX.Y.Z)                            │  ← B① Заголовок
│ Built-in DataWorks Official Skills                      │  ← B② Подзаголовок
│ Qwen OAuth | qwen-coder ( /model для переключения)       │  ← B③ Состояние
│ ~/projects/example                                      │  ← B④ Путь
└─────────────────────────────────────────────────────────┘

Ограничения:

  • Допускается только одна строка. Символы перевода строки и другие управляющие байты удаляются/сворачиваются в пробелы, чтобы избежать повреждения макета информационной панели при вставке.
  • Максимум 160 символов после очистки (более щадящее, чем для заголовка — слоганы / надписи “powered by” часто длиннее названия бренда).
  • Пусто (или пустая строка / полностью из пробелов) = оставить пустую строку spacer по умолчанию — обратная совместимость является поведением по умолчанию.
  • Подзаголовок не меняет поведение заблокированных строк; аутентификация, модель и рабочая директория всегда видны, независимо от наличия подзаголовка.

Замена ASCII art — встроенная строка

{
  "ui": {
    "customAsciiArt": "  ___  _    _  ____ \n / _ \\| |  / |/ _\\\n| |_| | |__| | __/\n \\___/|____|_|___|",
  },
}

В JSON-строке переводы строк обозначаются как \n. Этот ASCII-арт будет отображаться с текущим градиентом темы, как и стандартный логотип.

Нет под рукой ASCII art? Подойдёт любой внешний генератор, просто вставьте результат. Самый простой путь — figlet: npx figlet -f "ANSI Shadow" "xxxCode" > brand.txt, затем укажите customAsciiArt: { "path": "./brand.txt" } на этот файл. CLI не будет рендерить текст как ASCII art во время выполнения — причина ниже в разделе «Не входит в дизайн».

Замена ASCII art — внешний файл

{
  "ui": {
    "customAsciiArt": { "path": "./brand.txt" },
  },
}

Избегает экранирования больших многострочных строк в JSON. Правила разрешения пути:

  • Настройка уровня рабочей области: относительный путь разрешается относительно каталога .qwen/ рабочей области.
  • Пользовательская настройка: относительный путь разрешается относительно ~/.qwen/.
  • Абсолютные пути используются напрямую.
  • Файл читается только один раз при запуске, после очистки кешируется. Изменение файла во время сессии не вызовет перерендеринг — перезапустите CLI.

Замена ASCII art — адаптация по ширине

{
  "ui": {
    "customAsciiArt": {
      "small": "  ACME\n  ----",
      "large": { "path": "./brand-wide.txt" },
    },
  },
}

Если терминал достаточно широк, приоритетно используется large; в противном случае — small; если и это не помещается, столбец логотипа скрывается (текущая стратегия отката для двух столбцов). small и large могут быть как строкой, так и { path }. Любой уровень может отсутствовать: при отсутствии переходите к следующему уровню.

Насколько большим может быть логотип? — Бюджет ширины

У заголовка и art нет «жёсткого лимита количества символов», есть только бюджет ширины, определяемый количеством столбцов терминала, и абсолютный жёсткий лимит для предотвращения зависания макета из-за некорректного ввода:

ЭлементЛимит
Количество столбцов терминала при запускеСколько сообщит терминал пользователя.
Внешний отступ контейнера4 столбца (2 слева + 2 справа).
Расстояние между логотипом и панелью2 столбца.
Минимальная ширина информационной панели44 столбца (40 путь + рамка + внутренние отступы).
Доступная ширина для art одного уровня при рендерингестолбцов терминала − 4 − 2 − 44 = столбцов терминала − 50.
Жёсткий лимит для одного уровня art после очистки200 столбцов × 200 строк. При превышении обрезается и выдаётся предупреждение [BANNER].
Жёсткий лимит для customBannerTitle после очистки80 символов. При превышении обрезается и выдаётся предупреждение [BANNER].

Примерные лимиты логотипа для типичной ширины терминала:

Столбцов терминалаМаксимальная ширина логотипа для отображенияПрактическое значение
8030Большинство figlet “ANSI Shadow” букв 7–11 столбцов, максимум 3 символа.
10050ANSI Shadow для короткого слова (около 6 букв) или двух коротких слов стопкой.
12070Вполне достаточно для art с многострочным стекингом слов.
200150Помещается даже длинная строка в одну строку (например, ANSI Shadow полного имени продукта).

Два эмпирических правила при разработке art:

  1. Многословное имя бренда обычно невозможно отобразить на большинстве терминалов в виде одной строки ANSI Shadow. Каждая буква ANSI Shadow занимает около 7–9 столбцов, даже для имени бренда Custom Agent (12 символов) одна строка потребует около 95 столбцов art — на 100-колоночном терминале после размещения информационной панели уже не хватит. Либо разбивайте слова на стопку, либо используйте более узкий figlet-шрифт, либо используйте компактную однострочную декорацию, например ▶ Custom Agent ◀.
  2. Когда один уровень должен быть «красивым на широком экране» и «не умирать на узком», используйте адаптивную по ширине форму { small, large }. В следующем примере large — это многострочный art для стекинга на терминалах ≥ 104 столбцов, small — однострочная декорация шириной 16 столбцов, если она не помещается, столбец логотипа скрывается.
{
  "ui": {
    "customBannerTitle": "Custom Agent",
    "customAsciiArt": {
      "small": "▶ Custom Agent ◀",
      "large": { "path": "./banner-large.txt" },
    },
  },
}

В banner-large.txt поместите вывод ANSI Shadow с наложением (около 54 столбцов × 12 строк). Можно сгенерировать с помощью следующей команды:

( npx figlet -f "ANSI Shadow" CUSTOM
  npx figlet -f "ANSI Shadow" AGENT ) > banner-large.txt

Комбинация трёх элементов

{
  "ui": {
    "hideBanner": false,
    "customBannerTitle": "Acme CLI",
    "customAsciiArt": {
      "small": "  ACME\n  ----",
      "large": { "path": "./brand-wide.txt" },
    },
  },
}

Как проверить

  1. Сохраните settings.json, перезапустите qwen — разбор Banner выполняется только при запуске.
  2. Измените ширину терминала, убедитесь, что переключение small / large работает как ожидается, и что на очень узкой ширине столбец логотипа корректно скрывается.
  3. Если результат не соответствует ожиданиям, просмотрите ~/.qwen/debug/<sessionId>.txt (символическая ссылка latest.txt указывает на текущий сеанс), выполните grep [BANNER] — каждая мягкая ошибка выведет предупреждение с указанием причины.

Пятиэтапный алгоритм разбора выполняется один раз при загрузке настроек и повторно только при событии горячей перезагрузки настроек:

  1. Нормализация. Голые string или { path } преобразуются в { small: x, large: x }. Объекты { small, large } передаются без изменений.
  2. Поуровневый разбор. Для каждого AsciiArtSource:
    • Строка: используется напрямую.
    • { path }: синхронное чтение, с O_NOFOLLOW для защиты от захвата символических ссылок (в Windows понижается до обычного чтения только для чтения — эта константа не предоставляется), лимит 64 КБ. Относительный путь разрешается относительно каталога файла настроек, к которому принадлежит: настройка рабочей области — относительно .qwen/ рабочей области, пользовательская настройка — относительно ~/.qwen/. При ошибке чтения — [BANNER] warn, этот уровень возвращается к значению по умолчанию.
  3. Очистка. Специализированный stripper для Banner: удаляет управляющие последовательности OSC / CSI / SS2 / SS3, заменяет остальные управляющие байты C0 / C1 (включая DEL) пробелами, при этом сохраняет \n, чтобы многострочный ASCII art выжил. После обрезания завершающих пробелов каждой строки, обрезается до 200 строк × 200 столбцов; остальное обрезается и выводится предупреждение [BANNER].
  4. Выбор уровня при рендеринге. В Header.tsx, имея разобранные small и large, на основе существующего бюджета ширины (availableTerminalWidth ≥ logoWidth + logoGap + minInfoPanelWidth):
    • Если large помещается, предпочтительнее large.
    • Иначе, если small помещается, откат на small.
    • Иначе, если пользователь предоставил custom art, столбец логотипа просто скрывается (используется ветвь showLogo = false) — в этом случае откат на встроенный логотип QWEN мог бы незаметно нарушить белые метки на узком терминале. Информационная панель продолжает рендериться.
    • Иначе (пользователь вообще не предоставил custom art) откат на shortAsciiLogo, с проверкой шлюза ширины для логотипа по умолчанию.
  5. Запасной вариант. Если оба уровня из-за мягкой ошибки (отсутствие файла, пусто после очистки, некорректная конфигурация) в итоге оказываются пустыми или недопустимыми, рендерится shortAsciiLogo как если бы не было кастомизации, с обработкой по шлюзу ширины для логотипа по умолчанию. CLI ни в коем случае не должен падать из-за ошибки конфигурации Banner.

Псевдокод выбора уровня:

function pickTier(
  small: string | undefined,
  large: string | undefined,
  availableWidth: number,
  logoGap: number,
  minInfoPanelWidth: number,
): string | undefined {
  for (const candidate of [large, small]) {
    if (!candidate) continue;
    const w = getAsciiArtWidth(candidate);
    if (availableWidth >= w + logoGap + minInfoPanelWidth) {
      return candidate;
    }
  }
  return undefined; // Скрыть столбец логотипа
}

Парсер отвечает за нормализацию, чтение файлов, очистку и кеширование, описанные в «Конвейере разбора» выше. Вызывается один раз при запуске CLI и повторно при событии горячей перезагрузки настроек. Сравнение альтернатив

Ниже приведены 5 форм, которые были оценены ранее, чтобы помочь будущим разработчикам понять дизайн-пространство и при необходимости пересмотреть его.

Вариант 1 — три плоских поля (рекомендуется, полностью соответствует issue)

{
  "ui": {
    "customAsciiArt": "...", // string | {path} | {small,large}
    "customBannerTitle": "Acme CLI",
    "hideBanner": false,
  },
}
  • Эффект: Минимальный пользовательский интерфейс, полностью соответствует описанию issue.
  • Плюсы: Нулевая кривая обучения; документировать очень легко; совместимость с существующими плоскими полями ui.* (hideTips, customWittyPhrases и т.д.).
  • Минусы: Три семантически связанных ключа разбросаны по верхнему уровню ui; если в будущем появятся новые переключатели, связанные с banner (градиент, подзаголовок и т.д.), придётся добавлять братские поля в ui, а не в естественную группировку.

Вариант 2 — вложенное пространство имён ui.banner

{
  "ui": {
    "banner": {
      "hide": false,
      "title": "Acme CLI",
      "asciiArt": { "path": "./brand.txt" },
    },
  },
}
  • Эффект: Возможности такие же, как в варианте 1, но агрегированы по функциональности.
  • Плюсы: Будущие переключатели, специфичные для banner, получат чистое пространство имён; лучшая обнаруживаемость через /settings.
  • Минусы: Не полностью совпадает с изначальной записью issue; существующие настройки UI в основном плоские (вложены только ui.accessibility и ui.statusLine), что снижает согласованность; добавляет ещё один уровень, который нужно запоминать пользователю.

Подход 3 — Пресеты banner profile + переопределение слотов

{
  "ui": {
    "bannerProfile": "minimal" | "default" | "branded" | "hidden",
    "banner": { /* переопределение слотов в режиме 'branded' */ }
  }
}
  • Эффект: пользователь выбирает из именованных пресетов; продвинутые пользователи переопределяют конкретные слоты поверх выбранного пресета.
  • Плюсы: лучший опыт адаптации (onboarding); пресеты могут поставляться самим CLI.
  • Минусы: значительно возрастает сложность; пресеты — это долгосрочные обязательства по обслуживанию; в issue запрашивалась открытая кастомизация, а не курирование контента.

Подход 4 — Шаблон строки для всего баннера

{
  "ui": {
    "bannerTemplate": "{{logo}}\n>_ {{title}} ({{version}})\n{{auth}} | {{model}}\n{{path}}",
  },
}
  • Эффект: единый свободный шаблон с интерполяцией по полям, защищённым от изменений.
  • Плюсы: максимальная гибкость для нестандартных макетов.
  • Минусы: ответственность за макет перекладывается на пользователя; теряется устойчивость двухколоночного рендера Ink при разной ширине терминала; легко создать шаблон, ломающийся на узком терминале; слишком большая поверхность изменений ради сомнительной выгоды.

Подход 5 — Плагин / API-хук

Предоставить хук для отрисовки баннера через систему расширений.

  • Эффект: кастомизация на уровне кода; расширения могут рендерить произвольное содержимое.
  • Плюсы: максимальные возможности; предприятия могут упаковать целый брендовый плагин.
  • Минусы: огромная поверхность API; произвольный рендеринг в терминале требует проверки безопасности; полный овердизайн для данного issue.

Рекомендуемое заключение

Принять подход 1. Он напрямую удовлетворяет issue, соответствует существующему стилю ui.* и не привязывает пространство имён, пока мы точно не знаем, какие ещё поля, специфичные для баннера, могут появиться. Если в будущем начнут накапливаться подобные поля, миграция на подход 2 будет аддитивной — ui.banner.title и ui.customBannerTitle могут сосуществовать в период устаревания.

Безопасность и обработка ошибок

Пользовательское содержимое баннера выводится в терминал буквально, а в режиме path ещё и считывается с диска. Оба пути уязвимы при загрузке вредоносных или поддельных настроек. Та же модель угроз, что и для функции session-title, применима и здесь.

АспектМеры защиты
ANSI / OSC-8 / CSI-инъекции в ASCII-art / заголовке / подзаголовкеСпециализированный stripper для баннера (sanitizeArt / sanitizeSingleLine): удаляет управляющие последовательности OSC/CSI/SS2/SS3, заменяет остальные C0/C1 (включая DEL) пробелами. Применяется до рендеринга и до записи в кэш.
Заморозка запуска из-за огромного файлаЖёсткий лимит на чтение файла — 64 КБ.
Заморозка макета из-за патологического ASCII-artМаксимум 200 строк × 200 столбцов на результат парсинга; превышение обрезается + предупреждение [BANNER].
Подмена пути через символическую ссылкуЧтение файла с флагом O_NOFOLLOW (на Windows — только чтение; константа не экспортируется).
Файл отсутствует или нечитаемОтлов → предупреждение [BANNER] → откат к стандартному; никогда не вываливается в интерфейс.
Заголовок / подзаголовок содержат перевод строки или слишком длинныПереносы строк заменяются пробелами, обрезается до 80 (заголовок) / 160 (подзаголовок) символов.
Влияние ненадёжной рабочей области на рендеринг или чтение файлаЕсли settings.isTrusted равно false, парсер полностью пропускает settings.workspace (согласовано с шлюзом доверия settings.merged).
Состояние гонки при горячей перезагрузке настроекРезультаты парсинга мемоизируются по источнику (путь или строка) в пределах каждого вызова; при reload парсер запускается заново и перечитывает затронутые файлы.

Сводка по сбоям: все мягкие сбои в итоге приводят к shortAsciiLogo (или фиксированному заголовку по умолчанию) + одной строке отладочного лога warn. Никакая ветка не должна приводить к жёсткому сбою (выбрасыванию исключения).

План верификации

Будущий PR с реализацией должен пройти следующие сквозные проверки:

  1. В ~/.qwen/settings.json установить customBannerTitle: "Acme CLI" и встроенный customAsciiArt → после запуска qwen отображается новый заголовок и новый ASCII-art; суффикс версии всё ещё присутствует.
  2. Установить customBannerSubtitle: "Built-in Acme Skills" → строка подзаголовка появляется второстепенным цветом текста между заголовком и строкой «аутентификация / модель»; аутентификация, модель и путь всё ещё видны. Убрать настройку — возвращается пустая строка-разделитель (обратная совместимость).
  3. Установить hideBanner: true → qwen запускается без баннера; подсказки и основной контент отображаются как обычно.
  4. В workspace settings.json установить customAsciiArt: { "path": "./brand.txt" }, brand.txt находится в той же папке .qwen/ → при открытии рабочей области загружается с диска.
  5. customAsciiArt: { "small": "...", "large": "..." } → при изменении ширины терминала на широкую / среднюю / узкую: на широкой — large, на средней — small, на узкой — колонка логотипа скрывается; информационная панель всегда видна.
  6. В customBannerTitle и customBannerSubtitle внедрить \x1b[31mhostile → оба поля рендерятся как буквальный текст, не интерпретируются как красный цвет.
  7. path указывает на несуществующий файл → CLI запускается нормально; в ~/.qwen/debug/<sessionId>.txt появляется предупреждение [BANNER]; рендерится стандартный art.
  8. Открыть рабочее дерево при выключенном доверии к рабочей области → workspace-предоставленный customAsciiArt (включая { path }) игнорируется бесшумно; настройки уровня пользователя по-прежнему действуют.
